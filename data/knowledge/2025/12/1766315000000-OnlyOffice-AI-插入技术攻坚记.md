# 🚀 OnlyOffice AI 插件：用“广播通道”穿透 API 封锁

在构建 LearnHub 个人知识库的过程中，我们遇到了一个极具挑战性的技术难题：**如何在 OnlyOffice 社区版中实现 AI 文本的自动插入？**

这不仅是一次简单的 API 调用，更是一场与浏览器安全策略和厂商商业限制的“斗法”。

## 1. 绝望的“禁区”：OnlyOffice 社区版的限制

当我们满怀信心尝试调用 `editor.callCommand` 时，控制台冰冷的报错给了我们当头一棒。

*   **商业枷锁**：OnlyOffice 官方为了区分收费版，在**社区版（Community Edition）**中锁死了从外部网页直接调用自动化命令的权限。
*   **Iframe 牢笼**：编辑器运行在 Docker 容器的 `8888` 端口，而我们的主页面在 `3000` 端口。浏览器强大的**同源策略（SOP）**像一堵厚墙，阻断了 `postMessage` 的跨端口指挥。

## 2. 破局思路：寻找“视觉盲区”

常规的 API 调用走不通，我们必须另辟蹊径。我们发现了一个关键的**技术漏洞**（或者说特性）：

> **插件虽然运行在 8888 端口的 Iframe 里，但它的源代码地址却指向 3000 端口。**

这意味着：**主页面（3000）和插件（3000）其实是“亲兄弟”！** 它们在浏览器眼中属于同一个源。

## 3. 黑科技登场：BroadcastChannel (广播通道)

既然是亲兄弟，就不需要通过“中介”（OnlyOffice 的 API）传话了，直接用对讲机喊话就行！

### 核心方案：
1.  **主页面（发送端）**：不再尝试去“控制”编辑器，而是直接在浏览器底层打开一个名为 `onlyoffice-ai-channel` 的广播频道。
2.  **插件（接收端）**：潜伏在编辑器内部的自定义插件，同样打开这个频道，戴上耳机时刻监听。
3.  **穿透**：由于同源，广播消息完美绕过了 OnlyOffice 的 Iframe 隔离和 API 限制。

## 4. 关键代码实现

### 主页面：发射信号
```typescript
const channel = new BroadcastChannel('onlyoffice-ai-channel');
channel.postMessage({
  type: "ai-insert-text",
  text: aiText
});
```

### 插件 (code.js)：接收并注入
```javascript
const channel = new BroadcastChannel('onlyoffice-ai-channel');
channel.onmessage = function(event) {
    if (event.data.type === "ai-insert-text") {
        // 在编辑器“家里”直接执行插入，官方无法阻拦
        window.Asc.plugin.executeMethod("PasteText", [event.data.text]);
    }
};
```

## 5. 战果与启示

最终，我们成功实现了在**不花一分钱**、**不更换版本**的情况下，让 AI 文本顺畅地注入到 Word 文档中。

### 技术总结：
*   **不要被文档限制思维**：官方文档说“不支持”往往指的是官方路径，浏览器的底层能力（如 BroadcastChannel）经常能提供意想不到的解法。
*   **同源才是王道**：在处理复杂的 Iframe 通信时，保持代码同源是突破安全限制的最强武器。
*   **优雅的降级**：我们也保留了剪贴板方案作为终极兜底，确保了极致的用户体验。
